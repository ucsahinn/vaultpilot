# Sharing Screen

The topbar `?` opens this page from the Sharing screen. Use it to share selected records or external packages with clear expiry, recipient and usage limits.

## Work Here

- Share the minimum record set for the recipient's task.
- Set expiry and usage limits before creating an external package.
- Confirm the recipient path privately; do not send passphrases in the same channel as package files.

## Screen States

| State | Operator response |
| --- | --- |
| Draft share | Verify records, recipient and expiry before package creation. |
| Active share | Track usage and revoke when the task is complete. |
| Failed open | Use the external-share KB before sending a replacement package. |

## Before You Act

- Confirm the exact recipient, record set, expiry and maximum-open limits before sending.
- Use the narrowest package that solves the request; do not share entire vault context by convenience.
- Record the handoff purpose and revocation expectation before leaving the screen.

## Safe Evidence

- Safe to share: package state, expiry policy, max-open policy, error code and redacted recipient type.
- Keep private: package JSON, passphrases, recipient secrets, protected file names and decrypted contents.
- Use private support if delivery proof includes real recipient addresses, customer labels or package metadata.

## Operator Notes

Public issues may describe the sharing workflow, but must not include package contents, recipient identities, passphrases or unredacted links.

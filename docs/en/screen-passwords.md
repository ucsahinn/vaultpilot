# Passwords Screen

The topbar `?` opens this page from the Passwords screen. Use it for login records that belong to approved systems, shared operational accounts or individual service accounts.

## Work Here

- Verify the URL, owner, username, category and tags before revealing or copying.
- Use reveal and copy actions intentionally; both should leave useful audit evidence.
- Rotate weak, stale or exposed records through the owning system first, then update the vault record.

## Screen States

| State | Operator response |
| --- | --- |
| Empty result | Clear filters before assuming a record is missing. |
| Needs rotation | Confirm owner and dependency before changing the value. |
| Access denied | Request vault assignment through the normal owner workflow. |

## Before You Act

- Verify the selected vault, URL and username belong together before copy or fill actions.
- Check owner, last review and rotation hints before editing a record used by others.
- Prefer notes that describe the record role, not the plaintext value.

## Safe Evidence

- Safe to share: record category, vault assignment state, rotation status, access-denied code and redacted owner role.
- Keep private: plaintext passwords, usernames paired with target URLs, screenshots showing values, autofill contents and clipboard transcripts.
- Treat any public exposure of a password value as compromised and rotate through the owning system first.

## Operator Notes

Never publish plaintext passwords, screenshots showing values, browser autofill contents or copied clipboard transcripts.
